## Mission

Keep bug state under one obvious authority:

- bug rows live in `public.bugs`
- evidence links live in `public.bug_evidence_links`
- mutations go through the bug tracker surface, not ad hoc SQL
- the registry/docs are read models; the bug tracker surface is the write authority

## Core Laws

1. The bug tracker owns bug writes.
2. Evidence must be explicit in `bug_evidence_links`, not hidden only in prose.
3. Never invent `run_id`, `receipt_id`, `verification_run_id`, or `healing_run_id`.
4. `FIXED` without validation evidence is not done.
5. Prefer canonical enums on input even if the runtime can normalize aliases.

## Workflow

### 1. Inspect Before Filing

- Search or list first to avoid duplicate noise.
- Start with `praxis workflow bugs search "<title>"` or `praxis workflow bugs list --severity P1`.

### 2. Classify The Bug

Pick explicit values for:

- `severity`: `P0`, `P1`, `P2`, `P3`
- `category`: `SCOPE`, `VERIFY`, `IMPORT`, `WIRING`, `ARCHITECTURE`, `RUNTIME`, `TEST`, `OTHER`
- `source_kind`: where the bug came from, such as `manual`, `dispatch`, or `observability_hub`
- `filed_by`: actual caller identity
- `decision_ref`: preferred when the filing came from a governed decision path

Use provenance fields only when they can be verified:

- `discovered_in_run_id`
- `discovered_in_receipt_id`
- `owner_ref`

### 3. File Through The Authority Surface

Prefer the stable CLI surface:

```text
praxis workflow bugs file --title "<title>" --severity <P0|P1|P2|P3> --category <category> --description "<description>" --filed-by "<actor>" --source-kind <source_kind>
```

When CLI needs raw field control, use the tool surface:

```text
praxis workflow tools call praxis_bugs --input-json '{"action":"file","title":"...","severity":"P1","category":"RUNTIME","description":"...","filed_by":"...","source_kind":"..."}' --yes
```

Do not:

- hand-write `INSERT INTO bugs`
- supply your own `bug_id`
- supply your own `bug_key`

Important runtime behavior:

- `bug_id` and `bug_key` are generated by `BugTracker.file_bug()`
- `status` starts as `OPEN`
- `priority` currently mirrors `severity`
- `summary` is currently derived from the first 200 characters of `description`

Front-load the description with the actual failure, impact, and blocker so the auto-derived summary is useful.

### 4. Attach Evidence Immediately

After filing, attach authoritative evidence with:

```text
praxis workflow bugs attach_evidence --bug-id BUG-... --evidence-kind receipt --evidence-ref receipt-... --evidence-role observed_in
```

Canonical evidence kinds:

- `receipt`
- `run`
- `verification_run`
- `healing_run`

Canonical evidence roles:

- `observed_in`
- `attempted_fix`
- `validates_fix`

Use the obvious mapping:

- discovery receipt or run -> `observed_in`
- repair attempt -> `attempted_fix`
- verification proving the fix -> `validates_fix`

### 5. Resolve Carefully

Only resolve with terminal statuses:

- `FIXED`
- `WONT_FIX`
- `DEFERRED`

Prefer the stable CLI surface:

```text
praxis workflow bugs resolve --bug-id BUG-... --status FIXED
```

For FIXED with automated verifier proof in one step:

```text
praxis workflow bugs resolve --bug-id BUG-... --status FIXED --verifier-ref verifier.job.python.pytest_file --inputs-json '{"path":"..."}'
```

Before resolving `FIXED`:

- attach at least one `validates_fix` evidence link
- make sure the verification reference is real

## Required Filing Packet

When asked to log a bug, produce or gather:

- duplicate check result
- title
- severity
- category
- description
- filed_by
- source_kind
- decision_ref if available
- discovered provenance refs if available
- evidence actions to attach right after filing

## Output Contract

When using this skill, return:

1. `Duplicate Check`
2. `Bug Classification`
3. `File Action`
4. `Evidence Actions`
5. `Resolution Gate`

## Reject These

- direct SQL writes to `bugs` or `bug_evidence_links` outside migrations/tests
- prose like "seen in logs" without an attached evidence link
- fake provenance ids
- `FIXED` with no validation evidence
- severity words like "high" or "critical" in the final packet when canonical `P0`-`P3` values are available
