# Moon UI Action Audit

First-pass source audit for the Moon UI action surface.

Scope:
- Frontend: `Code&DBs/Workflow/surfaces/app/src/moon/*`
- Shared frontend API shim: `Code&DBs/Workflow/surfaces/app/src/shared/buildController.ts`
- API handlers: `Code&DBs/Workflow/surfaces/api/handlers/workflow_query.py`
- Runtime mutation/dispatch path: `Code&DBs/Workflow/runtime/canonical_workflows.py`

Notes:
- "Actually does now" is based on current source behavior, not intended product behavior.
- Runtime-injected `/api/catalog` rows are partially data-driven; static/default actions are enumerated below, and dynamic catalog behavior is called out explicitly.

## Edge / Gate Actions

Important split-brain behavior:
- Clicking a gate in the Detail dock uses `handleApplyGate` and persists edge gate data through `build_graph`.
- Dragging a control item onto an edge uses `applyCatalogToEdge`, which calls `refineDefinition(...)` with a prose instruction if `payload.definition` exists, and no-ops for manual graph-only builds.

## Primary Cleanup Targets

1. Collapse to one action authority.
   Right now actions come from at least three places: frontend static catalog, backend `/api/catalog`, and dead presenter `actionOptions`.

2. Separate "assign step type" from "run side effect".
   Most node catalog actions only assign a route/title, but the UI reads like they perform the action immediately.

3. Unify gate behavior.
   Edge gates currently use two incompatible paths: `build_graph` mutation on click vs prose `refineDefinition` on drag/drop.

4. Fix manual-builder persistence.
   Graph-first flows can dispatch, but they cannot cleanly save draft from the Action dock until a non-empty definition exists.
